# 아키텍처

> **TL;DR** — 정적 파일 4개로 이뤄진 클라이언트 전용 사이트입니다. 브라우저가 `index.html`을 열면 `app.js`가 `fetch`로 데이터를 받아 DOM 카드로 그리고, 이후 카드 펼치기·알림 신청·모의 폴링 같은 상호작용을 모두 브라우저에서 처리합니다. 서버 사이드 로직은 없습니다.

## 전체 그림

빌드 단계도, 서버 코드도 없습니다. 저장소 루트의 정적 파일을 정적 서버가 그대로 내려주고, 모든 동작은 브라우저에서 일어납니다.

```
index.html  →  styles.css (모양)
            →  app.js (동작)  →  fetch(DATA_SOURCE) → data.json → 카드 렌더링
```

- `index.html` — 페이지 골격. 헤더(제목 + 갱신 시각 자리), 상태 메시지 영역, 빈 카드 컨테이너만 둡니다. **진료과 데이터는 마크업에 하드코딩하지 않습니다.**
- `app.js` — 데이터 로딩과 DOM 렌더링. 모든 동적 동작이 여기 모여 있습니다.
- `styles.css` — 반응형 카드 그리드 레이아웃과 카드 스타일.
- `data.json` — 실제 API 응답 형태를 모방한 더미 데이터.

세 파일(`index.html`/`app.js`/`styles.css`)의 역할과 코드 위치는 [components/frontend.md](components/frontend.md)에서 자세히 다룹니다.

## 데이터 흐름

1. 브라우저가 `index.html`을 로드하고, `defer`된 `app.js`가 파싱 후 실행됩니다.
2. `app.js`의 `loadDepartments()`가 상태를 "불러오는 중…"으로 표시하고 `fetch(DATA_SOURCE)`를 호출합니다.
3. 응답 JSON의 `departments` 배열을 모듈 수준 `latestDepartments`에 보관한 뒤 카드로, `updatedAt`을 헤더의 갱신 시각으로 렌더링합니다.
4. 렌더 직후 `startMockPolling()`이 한 번 등록되어, 이후 호출 진행과 알림을 클라이언트가 이어서 처리합니다.
5. 실패하면 `#status` 영역에 오류 메시지를 보여줍니다.

첫 렌더 이후의 동작은 사용자 상호작용과 타이머로 굴러갑니다.

- **카드 펼치기** — 카드를 클릭하거나 키보드로 활성화하면 그 카드의 대기자 명단(`waitingPatients`)이 마스킹된 채 토글됩니다.
- **알림 신청** — 카드의 입력란에 내 대기 번호(`ticketNumber`)를 넣고 신청하면 `notifySubscriptions` Map에 구독이 저장되고, 카드가 입력 UI 대신 순서 상태로 다시 그려집니다.
- **모의 폴링** — `setInterval`이 `POLL_INTERVAL_MS`(3초)마다 각 과의 `currentNumber`를 1 올리고 `waitingCount`를 1 내려 호출이 진행되는 상황을 흉내 냅니다. 매 주기마다 다시 렌더하고 `checkNotifications()`로 내 순서가 3 이하가 되는 첫 시점에 `alert()`를 한 번 띄웁니다.

## 핵심 설계 결정

- **데이터 접근의 단일 분리점.** 데이터 출처는 `app.js` 최상단의 `DATA_SOURCE` 상수 하나뿐입니다. 실제 백엔드 연동 시 이 한 줄만 엔드포인트 URL로 바꾸면 됩니다. 응답 형태가 `data.json`과 같다면 나머지 코드는 그대로 동작합니다.
- **진료과는 데이터로 정의.** 화면에 보이는 3개 과(안과·내과·소아과)는 코드가 아니라 `data.json`의 배열로 정의됩니다. 과를 추가·수정하려면 데이터만 바꾸면 됩니다.
- **빌드리스(build-less).** 의존성·번들러를 두지 않아 `index.html`을 그대로 열거나 간단한 정적 서버로 띄울 수 있습니다.
- **클라이언트에서 호출 진행을 모의.** `data.json`은 정적 파일이라 스스로 줄지 않으므로, 알림 데모를 위해 `currentNumber`/`waitingCount`를 클라이언트 타이머로 변경합니다. 실제 백엔드가 붙으면 모의 폴링을 실제 폴링/푸시로 대체합니다.
- **구독 상태는 메모리에만.** 알림 신청은 `notifySubscriptions` Map(세션 한정)에만 남고 `localStorage` 등으로 영속화하지 않습니다. 새로고침하면 초기화됩니다.

## 알아둘 점 (gotcha)

- `file://`로 직접 열면 일부 브라우저의 CORS 정책 때문에 로컬 `fetch`가 차단될 수 있습니다. 루트 `README.md`가 안내하는 대로 `python -m http.server` 같은 정적 서버로 여는 것을 권장합니다.
- 더미 데이터 구조가 실제 API 응답과 다르면 교체 시 필드 매핑을 손봐야 합니다. 필드명을 직관적으로 정해 차이를 최소화했습니다.
- 모의 폴링은 매 주기마다 카드를 전부 다시 그립니다. 아직 신청하지 않고 입력 중이던 대기 번호는 폴링 시점에 초기화될 수 있습니다.
- `waitingCount`가 0이 되면 그 과의 `currentNumber`는 더 이상 오르지 않아, 아직 발동하지 않은 구독은 더 진행되지 않습니다(시뮬레이션 한계).
